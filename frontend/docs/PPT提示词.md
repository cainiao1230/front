# 实训项目4分钟汇报提示词（智慧养老前端）

> 建议用 6-7 页，每页 30-40 秒。按需取舍，尽量口语化表达。

---

## 1. 项目背景（~35s）
- 老龄化加速：机构需要数字化提升床位与护理管理效率。
- 线下流程痛点：床位调配慢、任务分派不统一、信息通知不及时。
- 本项目定位：基于 Vue3 + Vite 的“智慧养老前端”，承接后端 API，提供运营看板与业务入口。

## 2. 目标与意义（~35s）
- 目标：搭建可用的前端管理台，打通“老人—床位—护理—消息—系统”闭环。
- 核心意义：提升信息透明度与响应速度，减少人工沟通成本；为后续数据驱动运营打基础。
- 技术意义：规范 REST 接口调用与权限路由，沉淀可复用组件与业务模式。

## 3. 业务逻辑概览（~45s）
- 用户登录与权限：基于路由 `meta.permissionCode` 做 RBAC 放行与标题设置。
- 入住审批→床位分配：`elderly` 申请审批后，走 `beds/allocate` 或 `beds/{id}/assign` 完成分配与历史留痕。
- 护理任务→执行与记录：`care/tasks` 创建与状态流转（pending/in_progress/completed），护理与用药记录闭环。
- 消息/公告/待办：`messages` 与 `notices` 推送信息；`todos` 个人任务拆解；仪表盘汇总反馈。
- 运营看板：`/home` 展示预警、指标卡、今日任务、楼层床位、待办列表，支持降级回退。

## 4. 主要功能模块（~55s）
- 仪表盘：`/api/dashboard/stats` 指标卡（老人、床位、任务、护理员），实时预警与任务筛选。
- 老人管理：列表/搜索/新增/详情/编辑/审批；支持分页筛选与入住状态管理。
- 床位管理：列表/楼层/历史/分配/释放；`floor-stats` 统计、`floor/{floor}/grouped` 房间分组。
- 护理与用药：任务创建、执行状态、护理记录 CRUD；用药记录与“已服用”标记。
- 消息与公告：消息列表、详情、未读数；公告发布/更新/删除（管理员）。
- 系统管理：用户/角色/权限树、操作日志；密码修改与状态变更。
- 待办中心：个人待办的创建/更新/完成，支持状态筛选。

## 5. 数据与接口现状（~35s）
- 接口清单：按模块约 80 个接口，覆盖 CRUD 与状态变更（详见 [API_INTERFACE_CHECKLIST.md](../API_INTERFACE_CHECKLIST.md)）。
- 首页楼层统计：理想主接口 `GET /api/beds/floor-stats`；失败则回退到 `GET /api/dashboard/stats`（仅总床位汇总）。
- 问题记录：当前楼层统计可能未实现/数据为空，导致“各楼层床位使用情况”显示 No Data（详见 [HOME_PAGE_FLOOR_STATS_ISSUE.md](../HOME_PAGE_FLOOR_STATS_ISSUE.md)）。

## 6. 研究不足与改进方向（~55s）
- 接口一致性与回退：需确保 `floor-stats` 实装并统一响应 `{ code, message, data }`；完善前端降级提示与日志。
- 数据质量与校验：补齐 `beds` 的 `floor/status` 规范；增加边界校验与空数据占位。
- 权限细化：补充角色-权限映射的批量操作与前端可视化权限诊断（已有 Debug 页可拓展）。
- 告警规则与实时性：定义预警等级与触发条件；考虑 WebSocket/轮询优化实时预警栏。
- 可视化与报表：楼层热力图、使用率趋势、任务完成率周/月报；导出与打印。
- 体验与性能：列表虚拟滚动、接口缓存与并发控制；移动端/平板适配。

## 7. 总结与展望（~30s）
- 本期已完成路由与核心模块打通，具备运营看板与业务闭环能力。
- 下一步：优先落地 `floor-stats` 与数据填充，完善 RBAC、预警和报表，形成可交付版本。
- 期望配合：后端接口一致性与数据准备，共同完成联调与验收。
